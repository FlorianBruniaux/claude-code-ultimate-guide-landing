> **For agentic workers:** REQUIRED SUB-SKILL: Use superpowers:subagent-driven-development (recommended) or superpowers:executing-plans to implement this plan task-by-task. Steps use checkbox (`- [ ]`) syntax for tracking.

**Goal:** Correct all on-page SEO issues identified in the June 2026 audit: meta descriptions, titles, H1/H2 duplicates, sitemap lastmod, and one new keyword-targeted page.

**Architecture:** 5 independent tracks with zero file overlap, each can run in a separate git worktree simultaneously. Tracks A-D are pure edits to existing files. Track E creates a new page.

**Tech Stack:** Astro 5, TypeScript, @astrojs/sitemap, pnpm, `pnpm build` to validate.

## Global Constraints

- All meta descriptions: 155 chars max (soft cap 160, target 155 to give Google breathing room)
- All title tags: 60 chars max (desktop display limit)
- No placeholder text in new content; every section must have real, accurate content
- `pnpm build` must pass with zero errors before committing each track
- One commit per track with message format `seo(<track>): <description>`
- Worktrees: each track works on its own branch `seo/track-<letter>` created from `main`

### Task D-1: Add per-URL lastmod dates to sitemap

Currently the `serialize()` function in `astro.config.mjs` never sets `lastmod`, so the Astro sitemap plugin defaults all 181 URLs to the build date. Google uses `lastmod` for crawl prioritization. Identical dates on all pages neutralizes this signal entirely.

The fix: add a `URL_DATES` mapping for known high-value pages. For guide/ pages (140+ URLs), default to the build date since they genuinely all rebuild together. For the 12 named landing pages, use their actual last-modified dates.

- [ ] **Step 1: Add the date mapping object to `astro.config.mjs`**

  Insert this constant after line 11 (after the `__dirname` declaration) and before the `getCoreGuideItems` function:

  ```js
  // Per-page lastmod dates for key landing pages.
  // Guide pages (140+) are omitted, they rebuild together and all share the build date.
  const PAGE_DATES = {
    'https://cc.bruniaux.com/': '2026-06-19',
    'https://cc.bruniaux.com/cheatsheet/': '2026-05-03',
    'https://cc.bruniaux.com/faq/': '2026-06-08',
    'https://cc.bruniaux.com/releases/': '2026-06-19',
    'https://cc.bruniaux.com/examples/': '2026-06-08',
    'https://cc.bruniaux.com/quiz/': '2026-06-08',
    'https://cc.bruniaux.com/security/': '2026-06-18',
    'https://cc.bruniaux.com/ecosystem/': '2026-03-13',
    'https://cc.bruniaux.com/methodologies/': '2026-04-01',
    'https://cc.bruniaux.com/learning/': '2026-04-15',
    'https://cc.bruniaux.com/team-metrics/': '2026-05-01',
    'https://cc.bruniaux.com/context-engineering/': '2026-05-15',
  }
  ```

- [ ] **Step 2: Update the `serialize()` function to use lastmod from the mapping**

  In the `sitemap({ serialize(item) { ... } })` block, add `lastmod` to every returned object.

  The serialize function currently returns objects like `{ ...item, priority: 1.0, changefreq: 'weekly' }`. Change each branch to also include `lastmod`:

  ```js
  serialize(item) {
    if (item.url.includes('/sitemap/')) return undefined
    if (item.url.includes('/ultimate-guidefr/')) return undefined

    // Normalize URL (remove trailing slash for lookup, then add back)
    const normalizedUrl = item.url.endsWith('/') ? item.url : item.url + '/'
    const lastmod = PAGE_DATES[normalizedUrl] ?? new Date().toISOString().split('T')[0]

    if (item.url === 'https://cc.bruniaux.com/') {
      return { ...item, priority: 1.0, changefreq: 'weekly', lastmod }
    }
    if (item.url.includes('/quiz/')) {
      return { ...item, priority: 0.9, changefreq: 'monthly', lastmod }
    }
    if (
      item.url.includes('/learning/') ||
      item.url.includes('/security/') ||
      item.url.includes('/diagrams/')
    ) {
      return { ...item, priority: 0.85, changefreq: 'weekly', lastmod }
    }
    if (
      item.url.includes('/cheatsheet/') ||
      item.url.includes('/examples/') ||
      item.url.includes('/faq/') ||
      item.url.includes('/compare/') ||
      item.url.includes('/releases/')
    ) {
      return { ...item, priority: 0.8, changefreq: 'monthly', lastmod }
    }
    if (item.url.includes('/guide/')) {
      return { ...item, priority: 0.85, changefreq: 'weekly', lastmod }
    }
    return { ...item, priority: 0.7, changefreq: 'monthly', lastmod }
  },
  ```

        <h2>What to Include</h2>

        <h3>Always include</h3>
        <ul>
          <li><strong>Tech stack</strong>: language versions, key frameworks, database. <em>"Node 22, TypeScript strict, PostgreSQL 16, tRPC."</em></li>
          <li><strong>Build commands</strong>: how to run, test, and build. Claude will run these when needed. <em>"pnpm dev / pnpm test / pnpm build"</em></li>
          <li><strong>Code conventions</strong>: linting rules, naming standards, file organization. Anything you enforce in CI.</li>
          <li><strong>Architecture decisions</strong>: folder structure, key patterns, what NOT to do. <em>"Never put business logic in API routes."</em></li>
          <li><strong>Repeated context</strong>: anything you find yourself typing every new session.</li>
        </ul>

        <h3>Never include</h3>
        <ul>
          <li>General coding best practices, Claude knows these already</li>
          <li>Content that belongs in README (setup instructions for humans)</li>
          <li>API documentation, link to it instead of pasting it</li>
          <li>Per-developer personal preferences, use CLAUDE.local.md</li>
          <li>Anything that changes every week, stale instructions confuse Claude</li>
        </ul>

      <section class="content-section">
        <h2>Anti-patterns That Hurt Performance</h2>

        <ul>
          <li><strong>Padding with generic rules</strong>: "Write clean code", "Follow SOLID principles". Claude knows these. You're just burning context tokens.</li>
          <li><strong>Pasting entire files</strong>: if you reference a file by path, Claude can read it on demand. Pasting it upfront means it's read every session regardless of relevance.</li>
          <li><strong>Stale instructions</strong>: a CLAUDE.md referencing deprecated patterns (old library versions, renamed commands) actively misleads Claude. Review quarterly.</li>
          <li><strong>One monolithic global CLAUDE.md</strong>: global CLAUDE.md grows fast. Split personal preferences (global) from project conventions (project-level) from personal local paths (CLAUDE.local.md).</li>
          <li><strong>Missing build commands</strong>: this is the most common omission. Without them, Claude either guesses (wrong) or asks every session.</li>
        </ul>
      </section>

      <section class="content-section">
        <h2>FAQ</h2>
        <details class="faq-item">
          <summary>What should I put in CLAUDE.md?</summary>
          <p>Tech stack, build commands, code conventions, architecture decisions, and anything you find yourself typing every new session. Not general best practices (Claude already knows them), not README content, not per-developer preferences (use CLAUDE.local.md).</p>
        </details>
        <details class="faq-item">
          <summary>How long should CLAUDE.md be?</summary>
          <p>Project-level: 50-200 lines is the sweet spot. Global: up to 400 lines is reasonable. Over 500 lines, you're probably including content that belongs elsewhere (README, API docs, architecture diagrams). Longer ≠ better, every line costs tokens on every session.</p>
        </details>
        <details class="faq-item">
          <summary>Does CLAUDE.md replace README?</summary>
          <p>No. README is for humans setting up the project. CLAUDE.md is for Claude's in-session context. They overlap slightly (both might list build commands) but serve different audiences. Don't duplicate them, if it's in README, just reference the file path in CLAUDE.md.</p>
        </details>
